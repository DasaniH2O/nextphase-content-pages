The Savee 3.0 Build · Free Resource · 2026-07-22

He shipped a native iOS app in 33 days by critiquing Claude’s output like a mockup.

The full breakdown of Andre do Amaral’s designer‑feedback workflow — the loop he says carried Savee 3.0 through 1,078 prompts to the App Store, plus a starter prompt so you can run the same loop on your own build today.

The workflow, broken down

Savee 3.0 is a from‑scratch native SwiftUI rebuild of the design‑inspiration platform Savee — live on the App Store now, requiring iOS 17.0+. By his own account, do Amaral ran 1,078 prompts over 33 days, and Claude Code wrote the Swift. The part worth copying is what he did instead of reviewing code: he treated every build like a design review and gave feedback in critique language, the way he would mark up a junior designer’s mockup.

The designer‑feedback loop · one cycle
1

Describe the screen, not the architecture

Prompt Claude Code with what the screen should look like and do. Claude writes the SwiftUI; judging it happens on the screen, not in the diff.

2

Build and look — on the actual device

Run the app and judge it with designer’s eyes: spacing, hierarchy, motion, tint. The screen is the code review.

3

Critique in design language, verbatim

His actual feedback style, quoted from his own account of the build:

“Move that two points left.”
“Lighter tint.”
“That corner radius is wrong.”
4

Claude translates taste into SwiftUI

A “point” is SwiftUI’s actual layout unit, so critique like “two points left” maps directly onto padding, offsets, tint values, and corner radii. The critique is the spec.

5

Repeat until it ships

1,078 prompts over 33 days by his count — roughly 33 cycles a day — then three App Store releases in week one.

↩  back to 1 — every cycle ends in a new build to look at

Steps and quotes from do Amaral’s public account of the build (X thread, fetched 2026‑07‑21); the 33‑day and prompt counts are his own numbers, not independently verified.

The critique‑to‑SwiftUI phrasebook

The three quoted critiques above are his. The rows below extend the same vocabulary so you can run the loop without knowing Swift — each phrase is concrete enough for Claude Code to act on without a follow‑up question.

Say thisWhat Claude changes
“Move that two points left”Layout position — padding, offset, alignment values
“Lighter tint”Color — opacity, tint shade, contrast against background
“That corner radius is wrong — try 12”Shape — corner radius on cards, buttons, thumbnails
“Double the spacing between these rows”Vertical rhythm — stack and list spacing values
“This animation should settle faster”Motion — spring response and damping parameters
“The title is shouting — one size down, keep the weight”Type scale — font size and weight tokens
“Tappable area feels tiny — expand the hit target, not the icon”Interaction — content shape and padding, icon untouched

Rows 4–7 are our extensions of his method, not his quotes.

Run the loop yourself — starter prompt

Paste this into Claude Code inside your Xcode project to set the working contract. Claude works officially inside Xcode now, so the whole loop stays native Swift end to end.

You are building [SCREEN] of my iOS app in SwiftUI.
I am a designer, not an engineer. Our working contract:

1. I describe screens; you write and run the Swift. Do not ask me
   to review code — I review the running screen.
2. I give feedback as design critique: points, tints, radii,
   spacing, type sizes, motion. Apply it directly. "Two points
   left" means exactly that.
3. If my critique is ambiguous, show me two variants side by side
   rather than asking a technical question.
4. Keep every change buildable. If something I ask for fights the
   platform (safe areas, dynamic type, accessibility), say so in
   one plain sentence and propose the native-feeling alternative.

First screen: [describe layout, content, and the feeling it
should have — reference an app whose feel you want to match].

Worked example, filled in: “First screen: a full‑bleed image grid, two columns, 2‑point gutters, infinite scroll. Tapping a tile opens it full screen with a spring. The feel: calm and dense, like a moodboard wall.”

Why the loop held for 33 days

Two conditions did real work here, and both are worth engineering into your own attempt. First, taste with authority: one person could look at a screen and rule instantly, with no committee round‑trip between critique and change. Second, a spec that already existed: do Amaral is Savee’s own designer‑founder rebuilding a product he had lived in for years — the requirements were in his head, so every prompt carried real product knowledge. If you have neither, expect the loop to run slower — budget the first days for writing down what each screen must do before you critique pixels.
