Field Guide · Prompt Engineering · as of 20 Jul 2026

When a smarter model ships, delete the examples — keep the instruction.

Anthropic cut Claude Code’s system prompt by about 80% on purpose. The reason inverts standard few-shot advice: concrete examples anchor a capable model to imitate them, and a more imaginative model does better with the rule than with your samples. Here is the before/after that shows what “trim” actually means — and the one condition under which you should not do it.

Newer models “want a smaller system prompt” — examples “tend to constrain it because it’s actually more imaginative than the examples we give it.” Thariq Shihipar, Anthropic (Claude Code team) — AI Engineer talk, YouTube 4sX_He5c4sI, ~16:30

The centerpiece · worked example (illustrative)

Same task, two prompts — a support-ticket classifier

Before · built for a weaker model

System: Classify the ticket as BILLING, BUG, or FEATURE. Examples: "I was charged twice" → BILLING "Refund never arrived" → BILLING "App crashes on upload" → BUG "White screen after login" → BUG "Add dark mode please" → FEATURE "Wish it synced to Notion" → FEATURE Follow the pattern exactly. Output one word.

~70 tokens of few-shot. The model pattern-matches to these six and mis-files anything that doesn’t rhyme with them.

After · trimmed for a smarter model

System: Classify the ticket as BILLING, BUG, or FEATURE. BILLING = money, charges, refunds, invoices, plan changes. BUG = something broken vs. intended. FEATURE = a request for new behavior. Output one word.

The six examples become one line of decision criteria. Fewer tokens, and edge cases route by the definition instead of by resemblance to a sample.

What changed: examples deleted, the rule kept and sharpened. That is the 80% pattern in miniature — steer with criteria and context, not with samples to copy.

The re-trim habit — run it on model-upgrade day

1
Find the few-shot blocks. Any “Example:”, “e.g. do it like this”, or sample input/output pair in your system prompt. These are the anchors.
2
Replace each with its rule. Ask what the examples were teaching and write that as one line of criteria. Keep hard constraints (format, safety, output shape) — trim the illustrations, not the instructions.
3
A/B on your own evals. Run trimmed vs. original on a held-out set. Keep the trim only if quality holds or rises. If it drops, this model still wants the examples — put them back.

One honest limitation

This is conditional, not universal. “Fewer examples” works because these particular newer models are constrained by examples — it is not a blanket rule that every prompt should be 80% shorter. Weaker and older models historically need the examples to hit the target at all. The move is to re-trim each time a materially smarter model ships, guarded by your evals — not to strip samples indiscriminately. Why examples hurt now

A capable model is more imaginative than the examples you’d hand it. Given samples, it copies them — anchoring to your six cases and losing the range it already has. Delete the direction, steer with context.
