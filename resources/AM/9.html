Field Guide · Agent Architecture · as of 20 Jul 2026

Stop building one agent in one loop. Make the agents nodes in a graph.

The plan–act–check loop is one worker grinding a circle. Graph orchestration makes specialized agents into nodes, wired by explicit edges over one shared state — so work branches and runs in parallel. Below is a runnable LangGraph starter that fans out to two workers at once and loops back only if the review fails. Paste it and run it today.

The centerpiece · the shape you’re building

Fan-out, join, and a guarded loop — one shared state

Two workers run in the same superstep, not one after the other. The loop is the honest part — graphs still loop; they just also branch.

The starter — pip install langgraph

# LangGraph 1.0 — nodes, edges, one shared state. As of 20 Jul 2026. from typing import TypedDict, Annotated import operator from langgraph.graph import StateGraph, START, END # 1) SHARED STATE — every node reads/writes this one dict. # Annotated + operator.add MERGES parallel writes instead of clobbering. class State(TypedDict): topic: str findings: Annotated[list[str], operator.add] tries: int verdict: str # 2) NODES — each is a plain function: state in, partial state out. def plan(s): return {"tries": s["tries"] + 1} def research(s): return {"findings": [f"research on {s['topic']}"]} def draft(s): return {"findings": [f"draft on {s['topic']}"]} def review(s): ok = len(s["findings"]) >= 2 return {"verdict": "PASS" if ok else "FAIL"} # 3) EDGES — wire the shape. Two edges out of plan = parallel fan-out. g = StateGraph(State) for name, fn in [("plan",plan),("research",research),("draft",draft),("review",review)]: g.add_node(name, fn) g.add_edge(START, "plan") g.add_edge("plan", "research") # these two run g.add_edge("plan", "draft") # in the SAME superstep g.add_edge("research", "review") g.add_edge("draft", "review") # join: review waits for both # 4) THE GUARDED LOOP — pass ends it, fail loops back (capped). def route(s): if s["verdict"] == "PASS" or s["tries"] >= 3: return END return "plan" g.add_conditional_edges("review", route, {"plan": "plan", END: END}) app = g.compile() print(app.invoke({"topic": "prompt caching", "findings": [], "tries": 0, "verdict": ""}))

Run it: two workers fire together, review joins their merged findings, and the conditional edge either ends or loops — capped at 3 tries so it can’t spin forever. Swap the function bodies for real LLM/tool calls.

Make it real — three swaps

1
Nodes → agents. Replace each function body with an LLM or tool call. A node can itself be a full plan–act loop — that’s the point, loops live inside nodes.
2
Add workers to fan out wider. Every extra add_edge("plan", …) is one more agent running in the same parallel step. The reducer merges their writes.
3
Route on real signals. Have route branch on a scored review, a tool result, or a confidence threshold — not just a length check.

One honest limitation

“Loop vs. graph” is partly a story. LangGraph graphs still support loops — the conditional edge above is a loop. So loops didn’t die; they got composed into graphs. Don’t oversell the shift. The real, practical win is parallelism: if you’re running one agent in one sequential loop, you’re leaving concurrent work on the table. A single linear task gains nothing from a graph — reach for it when steps genuinely branch.
